## Context

Story 2159177 requests a styling and interaction pass for a rich text editor component. The current repository does not expose an obvious rich text editor implementation, so implementation must start by locating the editor source or adding the missing integration point before applying visual changes.

The app is a Next.js 16 / React 19 project using Tailwind CSS 4, shadcn-style UI primitives, and a green primary color token that maps to Studica branding. Any editor changes should fit those existing styling patterns and avoid introducing backend or data-model work.

## Goals / Non-Goals

**Goals:**

- Make the editor toolbar, dropdown, selected states, focus states, compact height, shortcuts, and link color match the story requirements.
- Split implementation into small tasks so visual styling, editor behavior, and verification can be handled separately.
- Prefer existing UI tokens and component primitives for focus rings, borders, shadows, dropdown surfaces, and icon buttons.
- Keep behavior accessible for keyboard users and screen readers.

**Non-Goals:**

- Redesign the full page or product workflow around the editor.
- Change saved editor content structure unless the existing editor library requires it for shortcuts or link styling.
- Add toolbar dividers before the revised visual hierarchy has been evaluated.
- Replace the editor library solely for icon or styling reasons.

## Decisions

- Discover the current editor integration before coding visual changes.
  - Rationale: The repository currently has no obvious text editor component, so the first implementation task must identify whether the editor lives in another branch, package, route, or planned component.
  - Alternative considered: Write generic CSS immediately. That is risky because editor libraries expose different DOM structures, command APIs, selected-state attributes, and shortcut hooks.

- Treat `rich-text-editor-ux` as one capability with separate implementation tasks.
  - Rationale: The requested changes are all about the same user-facing editor experience, but they are easier to implement and verify when split into toolbar icons, selected/focus states, dropdown styling, compact layout, shortcuts, and link color.
  - Alternative considered: Create separate OpenSpec capabilities for every visual item. That would over-fragment a single component styling story.

- Use existing design tokens for selected and focused states.
  - Rationale: Studica green is already represented by the app's primary token, and focus/outline behavior should stay consistent with the existing shadcn-style components.
  - Alternative considered: Hard-code exact color values. That would make the editor harder to maintain if the design system changes.

- Prefer library-supported editor commands and shortcuts.
  - Rationale: Markdown-like list shortcuts depend on how the editor parses input and manages document state. Built-in extensions or plugins are more reliable than manual keypress parsing.
  - Alternative considered: Hand-roll keyboard listeners for `-` and `1.`. That can conflict with IME input, undo history, selection handling, and accessibility.

- Keep the font dropdown as a dropdown first, with an icon popover as fallback.
  - Rationale: The story prioritizes a dropdown styled like Studica with a green arrow. The `AA` icon popover is explicitly second priority if dropdown styling or placement cannot meet the requirement.
  - Alternative considered: Convert directly to an icon popover. That would skip the preferred design direction.

## Risks / Trade-offs

- [Risk] The editor component is absent from this repository or generated by another package. -> Mitigation: Start implementation with a discovery task and document the actual component path before changing styling.
- [Risk] Font Awesome regular icons may not already be installed. -> Mitigation: Prefer existing icon dependencies if they can visually match; otherwise add the smallest appropriate icon dependency or local icon wrapper after confirming project standards.
- [Risk] Toolbar selected-state styling may depend on editor-library-specific active-state APIs. -> Mitigation: Use the library's command state or data attributes rather than duplicating editor state in React.
- [Risk] Dropdown placement can regress on small screens. -> Mitigation: Verify placement above and below the toolbar at mobile and desktop widths.
- [Risk] Reducing minimum editor height can make existing content harder to edit. -> Mitigation: Reduce only the minimum height, preserve natural growth/scroll behavior, and verify content remains usable.
- [Risk] Shortcut support may not be available in the current editor. -> Mitigation: Implement only with supported extensions/plugins, or record it as not feasible with the chosen editor.
